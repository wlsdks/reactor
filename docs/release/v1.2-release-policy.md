# Reactor 1.2 Release Policy

Reactor `v1.2.0` is one coordinated product release across the `reactor` and
`reactor-admin` repositories. Neither repository may publish an intermediate
`v1.2.*` tag while the product-operations consolidation goal is active.

## Version and tag contract

- `reactor/pyproject.toml` and `reactor-admin/package.json` must both read
  `1.2.0` in the final release-candidate changes.
- Each repository receives its own annotated `v1.2.0` tag at the exact commit
  that passed its final local gates. Lightweight tags are not release tags.
- The tag annotation records the coordinated release name and the companion
  repository commit SHA.
- Package version changes, annotated tags, and GitHub Releases happen only
  after the final readiness aggregate is `passed` and every warning is reviewed.
- A failed, blocked, skipped, stale, or HEAD-mismatched readiness input forbids
  both tags and both GitHub Releases.

## GitHub Release contract

- Create one GitHub Release named `Reactor 1.2.0` from each repository's
  annotated `v1.2.0` tag.
- Reactor release notes link the reactor-admin release and summarize backend,
  protocol, eval, and release-evidence changes.
- reactor-admin release notes link the Reactor release and summarize the
  administrator workflow and browser/live-stack evidence.
- Both releases record the paired commit SHAs and the final readiness artifact
  SHA-256. Draft releases may be prepared, but they must not be published before
  the coordinated release decision.

## HEAD-bound readiness contract

`reactor-release-smoke-run --readiness-output` binds every input report to:

- the current and expected Git commit SHA;
- an ISO-8601 generation timestamp derived from the input artifact;
- a canonical SHA-256 of the input JSON;
- a SHA-256 over the ordered aggregate input manifest.

The command fails closed when HEAD differs from the requested commit, an input
timestamp is in the future, or an input exceeds the configured freshness window.
The default freshness window is six hours and can only be changed explicitly
with `--max-readiness-input-age-seconds`.
